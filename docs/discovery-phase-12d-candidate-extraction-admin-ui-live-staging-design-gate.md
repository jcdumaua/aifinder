# Phase 12D — Candidate Extraction Admin UI Live Staging Design Gate

## 1. Status

Phase 12D is a docs-only design gate.

This phase does not implement Admin UI live staging.

This phase does not modify UI components, API routes, helpers, tests, package files, migrations, or generated types.

This phase does not run live staging, live smoke, database commands, remote SQL, Supabase commands, or POST requests.

This phase does not enable dry_run: false.

This phase designs a future high-friction Admin UI flow for staging one candidate through the guarded API foundation introduced in Phase 12C.

## 3. Phase 12D Goal

The goal of Phase 12D is to design a future Admin UI flow that can request one manual candidate staging action only after a later implementation and verification gate.

The future UI must be:

- high-friction;
- manual;
- admin-only;
- single-candidate only;
- tied to one exact discovery source;
- tied to one exact discovery run;
- visibly separate from dry-run;
- visibly separate from public publishing;
- visibly separate from discovered_tools mutation;
- unable to pass raw candidate payloads;
- unable to pass live gate objects;
- unable to pass service-role behavior;
- unable to bypass server authorization.

## 4. Non-Goals

Phase 12D does not design or authorize:

- public publishing;
- public.tools insertion;
- discovered_tools insertion;
- bulk candidate staging;
- automatic staging;
- crawler-triggered staging;
- LLM-triggered staging;
- background worker staging;
- source-wide staging;
- run-wide batch staging;
- UI implementation;
- route implementation;
- database migration;
- audit schema change;
- live verification execution.

## 5. Required User Experience Shape

A future Admin UI live-staging control must not be a simple dry_run toggle.

Recommended shape:

1. Keep the existing dry-run panel as-is.
2. Add a separate future section named "Stage one candidate" or similar.
3. Use a visually distinct warning style.
4. Require trusted source and run context before rendering the action as available.
5. Display exact discovery_source_id.
6. Display exact discovery_run_id.
7. Display audit_correlation_id or explain it will be generated by the client and validated by the server.
8. Display max_candidates: 1.
9. Display source_scope: single_run.
10. Display candidate source as "server-selected / reviewed candidate" or "not available" until a trusted candidate input provider exists.
11. Disable the action unless all required confirmations pass.
12. Disable repeat submission after success until the run detail context is refreshed.

The future UI must never present live staging as equivalent to dry-run.

The future UI must never use a casual toggle such as "Live mode".

The future UI must never include "Publish to public tools" in the same action.

## 6. Required Confirmation Friction

A future UI live-staging flow should require all of the following before enabling the final button:

- admin session is active;
- trusted discovery source ID is present;
- trusted discovery run ID is present;
- candidate preview or selected candidate reference is present from a trusted server source;
- warning checkbox is checked;
- typed confirmation phrase matches exactly;
- max_candidates visibly equals 1;
- source_scope visibly equals single_run;
- dry_run visibly equals false only in the final confirmation summary;
- UI clearly says the action writes one staging row only;
- UI clearly says this does not publish to public tools;
- UI clearly says this does not write discovered_tools;
- UI clearly says this cannot be undone from the UI unless a future cleanup/review flow exists.

Recommended typed phrase:

Stage one candidate

The typed phrase is only a UI confirmation string.

The typed phrase is not a ChatGPT approval phrase.

The typed phrase must not be treated as sufficient to activate server-side live staging.

The server must still independently authorize or reject the live gate.

## 7. Candidate Preview Requirement

A future UI live staging action must show a candidate preview before confirmation.

The preview should include only sanitized fields such as:

- candidate name;
- candidate website URL;
- candidate category hint;
- candidate pricing hint;
- confidence bucket;
- evidence summary;
- source evidence locator;
- discovery source ID;
- discovery run ID;
- audit correlation ID.

The preview must not show:

- raw HTML;
- raw extracted page text;
- LLM prompt text;
- LLM raw output;
- cookies;
- credentials;
- API keys;
- CSRF tokens;
- service-role details;
- database errors;
- stack traces;
- unbounded evidence payloads.

If no trusted candidate preview exists, the live staging control must remain disabled.

## 8. Candidate Input Source Boundary

The future UI must not send arbitrary candidate payloads.

The future UI must not send normalized candidate objects.

The future UI must not send raw extraction output.

The future UI must not send raw HTML.

The future UI must not send LLM output.

The future UI must not send stageCandidate.

The future UI must not send liveStagingGate.

The future UI may send only bounded intent fields approved by the future implementation phase.

Allowed client intent fields may include:

- discovery_source_id;
- discovery_run_id;
- audit_correlation_id;
- invocation_reason;
- dry_run: false;
- max_candidates: 1;
- source_scope: single_run;
- schema_version.

Even these fields are not sufficient to stage a candidate unless the server creates and validates the live gate.

The exact trusted candidate input source remains TBD and requires a later design or implementation phase.

## 9. API Request Boundary

The future UI should use the existing candidate extraction invocation route only if the route is explicitly approved for UI live staging in a later phase.

Future request requirements:

- method POST;
- same-origin credentials;
- CSRF token fetched from /api/admin/csrf;
- Content-Type application/json;
- no client admin identity;
- no liveStagingGate;
- no stageCandidate;
- no raw candidate payload;
- no service-role behavior;
- no public publishing flag;
- no discovered_tools flag;
- no batch flag.

The future UI must handle these route outcomes safely:

- accepted live staging;
- validation rejection;
- unauthorized;
- CSRF failure;
- rate limited;
- live_invocation_not_enabled;
- live_staging_not_configured;
- live_staging_input_unavailable;
- live_staging_failed;
- internal failure.

The UI must not display:

## 11. Post-Success UI Behavior

After a successful future live staging response, the UI should:

- show staged count as 1;
- show candidate_status as staged if safely available;
- show no_public_write_confirmed;
- show no_discovered_write_confirmed;
- show audit correlation ID;
- disable the final staging button;
- require refresh or navigation before another staging attempt;
- link to a future candidate review queue only after that queue is separately implemented.

The UI must not automatically retry.

The UI must not automatically publish.

The UI must not offer a one-click "stage another" action after success.

## 12. Failure UI Behavior

On failure, the UI should show a safe, bounded status.

Failure handling requirements:

- unauthorized: tell admin to log in again;
- CSRF failure: tell admin to refresh and retry;
- rate limited: tell admin to wait;
- validation rejection: show safe rejection_code only;
- live_invocation_not_enabled: show that live staging is not enabled;
- live_staging_not_configured: show that live staging is not configured;
- live_staging_input_unavailable: show that candidate input is unavailable;
- live_staging_failed: show safe error_summary only if sanitized;
- internal failure: show generic failure with no raw details.

The UI must never expose stack traces or database details.

## 13. Placement Design

Recommended initial placement:

- inside the Discovery Runs review area;
- near the existing candidate extraction dry-run panel;
- below or after dry-run results;
- visible only when an exact run and source context are present.

The existing dry-run panel should remain the primary first step.

The future live-staging section should be a secondary action that depends on dry-run/review readiness.

A future candidate review queue may become the better long-term placement after it exists.

## 14. Responsive Design Requirements

Desktop:

- show source/run/candidate summary in a clear multi-column card;
- keep the final danger action separated from dry-run controls;
- display confirmation checklist and typed phrase input clearly.

Tablet:

- use stacked sections if width is limited;
- keep warning, candidate preview, and final action visible without horizontal scrolling;
- ensure typed phrase input and final button are easy to tap.

Mobile:

- stack everything vertically;
- keep warning copy short but clear;
- place the final action after all context and confirmations;
- avoid sticky destructive buttons;
- prevent accidental taps by requiring explicit typed phrase.

## 15. Accessibility Requirements

Future UI implementation must support:

- semantic headings;
- visible labels for checkbox and typed phrase input;
- clear button labels;
- disabled state explanation;
- keyboard navigation;
- focus management after submit;
- screen-reader-friendly status messages;
- no color-only warnings;
- adequate contrast;
- no auto-submitting on Enter unless the final action is focused and enabled.

## 16. Audit And Review Requirements

A future Admin UI live staging flow should be auditable.

Desired future audit events:

- live staging UI viewed;
- live staging confirmation started;
- live staging confirmation cancelled;
- live staging requested;
- live staging rejected;
- live staging accepted;
- candidate staged;
- post-staging review required;
- failure stop condition.

Known limitation:

Direct candidate-audit linkage remains TBD unless a future audit schema or event metadata design is approved.

Future audit data must be sanitized and must not include raw HTML, raw payloads, secrets, CSRF tokens, service-role details, stack traces, or credentials.

## 17. Cleanup Vs Retention Design

Admin UI live staging should not use automatic cleanup by default.

Recommended policy for future real Admin UI staging:

- retain staged candidate row for review;
- show staged status only;
- require separate review/approve/reject flow before public publishing;
- do not write public.tools;
- do not write discovered_tools;
- do not delete candidate automatically unless a future cleanup tool exists.

For future verification-only UI smoke, exact-ID cleanup may be appropriate, but it must be defined by a separate live verification gate.
