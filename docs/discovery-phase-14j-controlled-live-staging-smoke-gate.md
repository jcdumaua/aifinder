# Discovery Phase 14J — Controlled Live Staging Smoke Gate

## Status

Draft live-smoke gate only.

This phase does not execute a live smoke, does not create a candidate, does not run a browser staging action, does not run a database command, and does not perform any production mutation.

## Purpose

Phase 14J defines the exact safety gate for a future controlled live staging smoke after Phase 14I implemented the admin UI staged action.

The goal is to prepare one tightly bounded, human-approved live staging smoke that proves the full server-revalidated staging path can stage exactly one candidate for admin review without publishing anything.

This document is a gate and runbook. It is not approval to execute the smoke.

## Current completed prerequisites

The following phases are complete and pushed:

- Phase 14C-E — trusted source URL provider validation.
- Phase 14D — backend preview-revalidated live gate resolver.
- Phase 14E — admin UI live staging activation plan.
- Phase 14F — backend route resolver wiring plan.
- Phase 14G — backend preview-revalidated route resolver wiring implementation.
- Phase 14H — admin UI staged action design.
- Phase 14I — admin UI staged action implementation.

Current state after Phase 14I:

- Admin UI staged action exists.
- UI fetches CSRF before POST.
- UI posts only the bounded invocation request body.
- Backend route performs session, CSRF, rate-limit, field allowlist, and server-created resolver checks.
- Resolver revalidates preview server-side.
- Staging remains candidate-review only.
- No Phase 14J live smoke has been run yet.

## Non-goals

Phase 14J does not authorize:

- Running the live smoke.
- Clicking the UI staging button in production.
- Creating candidate rows.
- Running a database command.
- Running Supabase CLI.
- Publishing to public tools.
- Writing to `public.tools`.
- Writing to `discovered_tools`.
- Adding approval or publishing workflow.
- Adding bulk staging.
- Adding background automation.
- Adding crawler execution.
- Adding LLM extraction execution.
- Changing schema.
- Changing generated types.
- Changing backend route behavior.
- Changing UI implementation.

## Required exact approval before any live smoke

A future live staging smoke must not run unless the user gives this exact approval phrase:

`Approve run Phase 14J controlled live staging smoke`

Any other wording, including `approved, proceed`, authorizes planning/readiness only and does not authorize a live mutation.

## Smoke principle

The future smoke must prove only this:

> One accepted, reviewable, server-revalidated preview can stage exactly one candidate with `candidate_status = "staged"` while confirming no public `tools` write and no `discovered_tools` write.

The smoke must not prove publishing, approval, ranking, public search visibility, crawler automation, or LLM automation.

## Candidate scope

The smoke must use exactly one known safe preview context:

- One `discovery_source_id`.
- One `discovery_run_id`.
- One `audit_correlation_id`.
- One accepted preview with `previewStatus = "reviewable"`.
- One trusted HTTPS `sourceUrlSnapshot`.
- One HTTPS `candidateWebsiteUrl`.
- `sourceUrlSnapshot` and `candidateWebsiteUrl` must be different URLs.
- `max_candidates = 1`.
- `source_scope = "single_run"`.
- `dry_run = false`.

## Safe source requirements

The chosen source/run must satisfy:

- Source URL is public HTTPS.
- Source URL is not localhost, private IP, internal hostname, credentialed URL, or non-HTTP scheme.
- Source URL matches the preview artifact `sourceUrlSnapshot`.
- Source URL is not the same as the candidate website URL.
- Candidate website URL is public HTTPS.
- Preview artifact was generated by the controlled pipeline and is reviewable.
- Preview artifact is not stale compared to the run.
- Preview artifact has no blocking safety flags.

## Future live smoke request shape

The future live POST body must be exactly bounded to this shape:

```json
{
  "discovery_source_id": "<source uuid>",
  "discovery_run_id": "<run uuid>",
  "audit_correlation_id": "<audit correlation uuid>",
  "invocation_reason": "Admin requested staged candidate from reviewable preview.",
  "dry_run": false,
  "max_candidates": 1,
  "source_scope": "single_run",
  "schema_version": "candidate_extraction_invocation.v1"
}
```

The request must not include:

- `invoked_by_admin_user_id`
- `liveStagingGate`
- `getLiveStagingCandidate`
- `stageCandidate`
- `sourceUrl`
- `source_url`
- `sourceUrlSnapshot`
- `candidateWebsiteUrl`
- `candidate_website_url`
- `preview`
- `raw_payload`
- `public_tool_payload`
- `discovered_tool_payload`

## Future UI smoke path

Preferred smoke path:

1. Admin signs in.
2. Admin opens `/admin/discovery`.
3. Admin expands or opens the selected run review.
4. UI fetches candidate preview.
5. UI shows accepted reviewable preview.
6. UI shows `Stage candidate` action.
7. Admin opens confirmation dialog.
8. Admin confirms checkbox.
9. Admin types `STAGE CANDIDATE`.
10. Admin clicks `Stage candidate`.
11. UI obtains CSRF and posts bounded body.
12. Server revalidates preview.
13. Server stages exactly one candidate.
14. UI shows success state.

This UI path should be used only after exact approval.

## Future API smoke path

An API smoke path may be safer for the first live staging smoke if the admin session and CSRF can be obtained through existing authenticated browser or script patterns.

If API path is used, it must still:

- Use authenticated admin session.
- Use valid CSRF.
- Send only the bounded body.
- Capture full response.
- Confirm no public write.
- Confirm no discovered write.
- Avoid service-role direct insertion from the test harness.

## Success criteria

The future smoke passes only if all are true:

- Response is successful.
- Response has `accepted = true`.
- Response has `dry_run = false`.
- Response has `candidates_considered_count = 1`.
- Response has `candidates_staged_count = 1`.
- Response has `candidates_skipped_count = 0`.
- Response includes a candidate ID.
- Response has `candidate_status = "staged"` or equivalent staged confirmation.
- Response audit correlation matches the chosen smoke audit correlation ID.
- Response confirms `no_public_write_confirmed = true`.
- Response confirms `no_discovered_write_confirmed = true`.
- Safety flags include bounded live staging markers.
- No public directory page changes are observed.
- No publishing action occurs.

## Failure criteria

The future smoke must be treated as failed-safe if any are true:

- Response is rejected.
- Response has `live_invocation_not_enabled`.
- Response stages zero candidates.
- Response stages more than one candidate.
- Response does not confirm no public write.
- Response does not confirm no discovered write.
- Response audit correlation mismatches the chosen ID.
- UI displays ambiguous success.
- Any unexpected public or discovered write is detected.
- Any raw error, stack trace, secret, token, service role, raw HTML, or raw payload appears in UI.

A failed-safe result may still be acceptable from a security perspective, but it does not prove live staging success.

## Cleanup policy

Preferred cleanup policy:

- Leave the staged candidate record as a traceable smoke artifact only if it is clearly marked as a smoke/test candidate and does not appear publicly.
- Otherwise, document exact cleanup steps before the live smoke and execute cleanup only if separately approved.

No destructive cleanup command should be run without explicit approval.

## Audit and documentation requirements

The future smoke output must record:

- Date/time.
- Current commit.
- Selected source ID.
- Selected run ID.
- Selected audit correlation ID.
- Candidate ID returned.
- Candidate status returned.
- Response summary.
- Safety flags.
- Confirmation that no public write occurred.
- Confirmation that no discovered write occurred.
- Any cleanup decision.
- Final repo status.

A follow-up Phase 14K document should summarize the result.

## Abort conditions

Abort before live smoke if:

- Working tree is dirty.
- Latest commit is not Phase 14I or later approved commit.
- Baseline verification fails.
- Admin session cannot be validated.
- CSRF cannot be fetched.
- Preview is not accepted and reviewable.
- Source URL snapshot is missing, unsafe, or stale.
- Candidate website URL is missing or unsafe.
- Source and candidate URLs are the same.
- More than one candidate would be staged.
- Exact approval phrase has not been provided.

Phase 14J gate is complete when:

- This gate document is committed.

- No database command has been run.
- No source code has changed.
- No UI code has changed.
- No route code has changed.
- No schema migration has been added.
- `npm run check` passes.
- Gemini approves the controlled live staging smoke gate.
